## Description

Problem Summary
What the Customer Reported
A customer testing Microsoft Email Security safeguards saw multiple checks returning:
{
  "insurer": "Microsoft - Email Security",
  "errorMessage": "Integrator not configured",
  "status": "Not Available",
  "message": "Integrator not configured"
}
They had already added the required Azure AD Graph permissions (IdentityProvider.Read.All, Policy.Read.All, AuditLog.Read.All) and granted admin consent — yet these 9 checks still failed:
areAntiPhishingPoliciesConfigured, isAntiPhishingEnabled, isSMTPAuthDisabled, isBannerModeEnforced, isAppConsentRestricted, isAutoForwardDisabled, isEmailFilteringEnabled, isMacroBlockingEnabled, isAdminAuditLoggingEnabled
Adding permissions had nothing to do with the problem — the checks were broken before they even reached the API call.
Root Cause
The engine that runs checks (integration_utils.py) works like this:
Receives a check name (e.g. isAdminAuditLoggingEnabled)
Looks for it in the workflow section of the integrator's config JSON
If found → executes the workflow steps (API calls)
If not found in workflow AND not a top-level endpoint key → returns {"status": "Not Available", "message": "Integrator SRN {methodName} not configured"}
The Microsoft Email Security integrator config (ms365.json) was missing entries for all 9 failing checks:
7 checks had zero entries — not in workflow, not in retrievalTransformationArray — the engine hit the fallback "not configured" path immediately.
isAntiPhishingEnabled had a workflow entry, but it called getMailPolicies — an endpoint that was not defined anywhere in ms365.json. So it got past the workflow lookup but failed when trying to execute the step.
isAdminAuditLoggingEnabled (visible in the screenshot) was the same as the 7 — no workflow entry at all.
The permissions the customer added were correct and necessary — the integrator just didn't have the plumbing to use them yet.
What I Changed
All changes were made to integration_configs/emailsecurity/ms365.json in the Integration-Service repo.
Added workflow entries for all 9 checks, mapping each to an appropriate underlying method:

Added 4 new endpoint definitions:
getMailPolicies — Lambda PowerShell call (Get-AntiPhishPolicies.ps1) via the existing Lambda URL, same pattern as getExchangeTransportRules
getSMTPAuthConfig — Lambda PowerShell call (Get-SMTPAuthConfig.ps1)
getMalwarePolicies — Lambda PowerShell call (Get-MalwarePolicies.ps1)
getAuthorizationPolicy — Simple Graph API GET https://graph.microsoft.com/v1.0/policies/authorizationPolicy (this is what Policy.Read.All unlocks)
Added retrievalTransformationArray entries for all 9 checks pointing to the standard transformation script URL pattern in the Transformations repo.
What Still Needs to Be Done
