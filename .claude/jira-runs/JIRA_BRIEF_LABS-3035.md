# LABS-3035: Evaluate scores integration auth/connection failures (HTTP 401/403) as compliance failures (red), tanking posture and creating false gaps

## Description

The defect
During safeguard evaluate, when the vendor API call returns an authentication/connection error (most commonly HTTP 401 — bad/expired credentials), the pipeline does not recognize it as a connection failure. The error response is handed to the safeguard's transform(), which finds none of its expected keys, returns value: false, and the safeguard is rendered red — indistinguishable from a genuine compliance failure.
This is a correctness defect: "we can't authenticate to your tenant" is reported to the customer as "your security control failed."
Reproduction (observed in production)
Customer: Impelix. All Microsoft safeguards red since May 16, 2026, every one returning 401. The customer reported it as "Microsoft went red→green with no changes on our side" — the classic signature of an OAuth token / admin-consent expiry followed by a refresh. It went undetected for ~6 weeks because nothing alerts on it.
Steps to reproduce deterministically:
Point a safeguard's integration at credentials that are expired/revoked (or mock the vendor endpoint to return 401 Unauthorized).
Run evaluate for that safeguard.
Observed: safeguard = red; posture score drops; gap(s) created; no email; no task.
Expected: safeguard = grey ("connection failure"); score unchanged; customer notified; remediation task created.
Root cause
No status-code handling in transforms. All 18 transformations in fedramp-20x-checks/transformations/*.py do response.get("<key>", []) and never inspect HTTP status. A 401 body has none of the expected keys → empty → value:false.
Concrete example: fedramp-20x-checks/transformations/areiamprivilegesscoped.py — findings = response.get("Findings", []) → value: len(wildcard) == 0 is computed over an empty list of the wrong data.
value:false → red. The pipeline cannot distinguish "control failed" from "no valid data."
Red is scored against the customer. flux/apps/mosaic/src/app/(authenticated)/dashboard/components/posture-heatmap/use-posture-data.ts counts red into currentStatus and the score.
Impact
Customer trust: customers see false compliance failures and chase non-existent problems.
Score integrity: posture score and trend are corrupted by connection noise.
False gaps/requirements: downstream gap/requirement artifacts are generated from bad data.
Silent outages: a broken integration can persist for weeks (Impelix: ~6) with no signal.
Severity / Priority
High → posture accuracy is a core product promise and this misrepresents customer security state. Affects any integration that can return 401/403 (Microsoft Graph, Okta, MDR, AWS, etc.).
Acceptance criteria (this ticket = the agreed problem record + a failing test)
A reproduction test exists that mocks a 401 from a safeguard's integration and asserts the current (wrong) behavior: safeguard goes red, score drops. This test becomes the regression guard once the fix (sibling tickets) lands and flips it to expect grey + unchanged score.
